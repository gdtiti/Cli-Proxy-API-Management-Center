## 备选方案对比

### 方案 A（配置/收口优先）

- 做法: 以现有工作区 diff 为线索，确认 `.mono/monoConfig.json` 是否需要纳入仓库；如需要则补齐格式/换行并跑最小验证（`pnpm run lint` / `pnpm run build` / `pnpm run type-check`）。
- 收益: 改动面小、风险低，容易验收与提交。
- 代价: 若真实目标是 UI 功能/业务修复，则覆盖不足，需要追加阶段。
- 风险: `.mono` 配置可能是外部工具专用；误提交可能带来团队环境差异问题。

### 方案 B（功能/体验修复优先）

- 做法: 若目标是前端功能/体验修复（AI Providers / Quota / Usage / Auth Files 等），先明确具体页面与行为，再按模块落地并配套验证。
- 收益: 直接解决用户可感知问题。
- 代价: 可能涉及多文件/多模块，实施与回归成本更高。
- 风险: 需求未收敛时容易 scope creep；需要更严格的阶段切分与证据门禁。

## 锁定设计

- 入口模块:
  - 若是配置/工具类: `.mono/monoConfig.json`
  - 若是 UI 功能: 页面在 `src/pages/*`，路由在 `src/router/MainRoutes.tsx`
- 关键调用链: Page -> store（Zustand）-> `src/services/api/*` -> components
- 数据流: config/authFiles/quota/usage 等经由 store 聚合，页面订阅并渲染
- 错误处理: 以全局通知/确认流为主（具体实现按目标定位）
- 配置变更: 仅在被证实需要时修改，并避免引入环境特异配置

### 阶段 2（Execution + Acceptance：实现与证据门禁）

- 目标: 完成最小可验证实现 + 收集 fresh evidence + 进入验收给出结论

- 主要变更: 按批准范围修改代码或配置；回写 `技术说明.md` 的改动与验证记录
- 验证方式: `pnpm run lint` / `pnpm run build` / `pnpm run type-check`（按实际需要选）
- 完成标志: 至少一条 REQ 的证据成立，可给出 `accepted` / `conditionally accepted` / `rejected`
