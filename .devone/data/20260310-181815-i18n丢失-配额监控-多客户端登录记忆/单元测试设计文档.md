## 测试目标

- 需要被保护的核心行为:
  - i18n：关键菜单与新增 UI 文案在所有支持语言中均存在对应 key，避免回显 key/空白。
  - 安全边界：多客户端管理密钥不落入可读的 persist JSON（仅通过 secureStorage 加密存储）。
  - 登录后切换：切换客户端失败时不应触发“被强制登出/跳回登录页”的体验回归。

- 这轮必须覆盖的失败模式:
  - `t('nav.monitor')` 等 key 缺失导致侧边栏显示 `nav.monitor`。
  - `client_selector.*` / `client_management.*` 缺失导致新增 UI 回显 key。
  - 切换客户端时出现 401 导致全局 unauthorized 事件把用户踢回登录页。

## 单元测试设计

### UT-001：导航菜单 i18n key 存在性

- 目标:
  - 防止 `nav.monitor` / `nav.quota_management` 等关键菜单 key 缺失导致 UI 回显 key。
- 覆盖函数/模块:
  - `src/i18n/locales/zh-CN.json`
  - `src/i18n/locales/en.json`
  - `src/i18n/locales/ru.json`
- 输入:
  - 读取并解析 JSON。
- 预期输出:
  - 三个语言文件都包含 `nav.monitor` 与 `nav.quota_management`。
- 失败表现:
  - 测试失败并指出缺失 key。

### UT-002：多客户端 UI i18n key 存在性

- 目标:
  - 防止新增的“客户端选择/管理”入口因为缺 key 回显 `client_selector.*` / `client_management.*`。
- 覆盖函数/模块:
  - 同 UT-001 的三份 locale JSON。
- 输入:
  - 读取并解析 JSON。
- 预期输出:
  - 三个语言文件都包含最小集合：
    - `client_selector.label`
    - `client_selector.placeholder`
    - `client_management.title`、`client_management.saved_clients`、`client_management.empty`、`client_management.add_new`、`client_management.select`、`client_management.edit`、`client_management.delete`、`client_management.delete_confirm`
    - 以及表单相关 key（name/apiBase/key 的 label 与 placeholder）。
- 失败表现:
  - 测试失败并列出缺失 key。
