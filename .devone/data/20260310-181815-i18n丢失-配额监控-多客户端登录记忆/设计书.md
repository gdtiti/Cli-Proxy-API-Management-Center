## 方案摘要

- 推荐方案: 方案 A（最小修复 + 接线恢复）+ 补齐“登录后快速切换”
  - i18n：补齐 `nav.monitor` 等缺失 key；复核并保留 `nav.quota_management`。
  - 多客户端（登录页）：复用现有 `useClientCacheStore` + `secureStorage` 的实现，接入登录页 UI（选择器 + 管理弹窗），恢复“记忆 + 快速选中并登录”。
  - 多客户端（登录后）：在 Header 增加客户端切换入口，用户登录后可快速切换并触发页面刷新，避免显示旧环境数据。

- 推荐原因:
  - 当前仓库里多客户端相关 store/组件/设计文档已存在，主要缺口是“未接入页面 + i18n key 缺失”。
  - 改动面小、跨层冲击低（纯前端），验证路径清晰（本地启动 + 手工操作即可验收）。
  - 能用最少新抽象恢复用户已感知能力，符合回归修复优先级。

## 备选方案对比

### 方案 A（推荐）：最小修复 + 接线恢复

- 做法:
  - i18n：在 `src/i18n/locales/*` 的 `nav` 节点补齐 `monitor` 翻译。
  - 多客户端（登录页）：将 `ClientSelector` + `ClientManagementModal` 接入 `LoginPage.tsx`；登录成功后更新 `activeClientId/lastConnectedAt`。
  - 多客户端（登录后）：在 `MainLayout.tsx` 的 Header 接入客户端切换控件；切换时使用 `useAuthStore` 执行重连，并触发 `triggerHeaderRefresh()` 刷新当前页数据。
  - 补齐缺失的 `client_selector.*` / `client_management.*` i18n 文案。

- 收益:
  - 直接恢复用户能感知的缺失能力，变更面小。
  - 复用现有 store 与 secureStorage，加密边界已具备。

- 代价:
  - 需要补齐一批 i18n key，并在登录页做少量 UI 排版调整。

- 风险:
  - 切换连接属于高影响操作：需要避免 401 触发全局登出导致“切换失败被踢回登录页”的体验回归。
  - 切换后可能出现“页面仍显示旧环境数据”的状态残留，需要复用现有 Header refresh 机制做一次强刷新。

### 方案 B：重构为“客户端配置”为单一真相 + 登录后快速切换

- 做法:
  - 将 `useAuthStore` 的 `apiBase/managementKey` 与 `useClientCacheStore.activeClientId` 深度绑定。
  - 在 Header 内增加客户端切换控件，切换时自动清理缓存并重新连接；启用 `useClientKeyboardShortcuts`。

- 收益:
  - 真正意义上的“登录后快速切换”体验完整。

- 代价:
  - 改动会跨多个 store（auth/config/usage 等）与 UI，回归面更大。

- 风险:
  - 容易产生状态残留（切换后仍显示旧环境数据），必须增加更强的验证与回归。

## 锁定设计

- 入口模块:
  - 导航菜单：`src/components/layout/MainLayout.tsx`
  - 语言资源：`src/i18n/locales/zh-CN.json`、`src/i18n/locales/en.json`、`src/i18n/locales/ru.json`
  - 登录页：`src/pages/LoginPage.tsx`
  - 多客户端 store：`src/stores/useClientCacheStore.ts`
  - 多客户端 UI：`src/components/client/ClientSelector.tsx`、`src/components/client/ClientManagementModal.tsx`
  - （可选）快捷键：`src/stores/useClientKeyboardShortcuts.ts`

- 关键调用链（现有）：
  - `LoginPage` -> `useAuthStore.login(credentials)` -> `apiClient.setConfig({ apiBase, managementKey })` -> `useConfigStore.fetchConfig()` -> 登录成功/失败 UI 反馈。

- 数据流（计划接入）：
  - `useClientCacheStore` 持久化 `clients（不含密钥）/ activeClientId` 到 `localStorage(client-cache)`。
  - `secureStorage` 单独加密保存 `client-cache-<id>-key`。
  - `LoginPage`：选择客户端 -> 从 `getClientById()` 获取密钥 -> 填充输入框 -> 调用 `login()`。
  - 登录成功后：更新 `activeClientId` 与 `lastConnectedAt`（用于排序/提示）。

- 错误处理:
  - i18n：缺 key 时会回显 key；通过补齐 key 消除。
  - 登录/切换：沿用现有 `getLocalizedErrorMessage` + `showNotification`。

- 配置变更:
  - 不新增后端配置项；仅增加前端本地缓存（localStorage/secureStorage）使用。

## 分阶段计划

### 阶段 1（必备）：恢复菜单 i18n + 登录页多客户端闭环

- 目标:
  - 修复侧边栏「监控」菜单多语言缺失。
  - 恢复“多客户端登录记忆 + 登录页快速选中并登录”。

- 风险主题:
  - i18n key 缺失导致 UI 回显 key。
  - 多客户端接线涉及敏感密钥存储边界与登录流程兼容性。

- 主要变更:
  - `src/i18n/locales/*`：补齐 `nav.monitor`；补齐 `client_selector.*`、`client_management.*`。
  - `src/pages/LoginPage.tsx`：接入 `ClientSelector` 与 `ClientManagementModal`；选择客户端后填充输入；登录成功后更新 `useClientCacheStore` 状态。

- 验证方式:
  - 本地启动 `pnpm dev`，切换语言验证菜单翻译。
  - 登录页：新增 2~3 个客户端，切换选择并能登录；刷新页面后仍存在。
  - 检查 localStorage：`client-cache` 中 `clients[*].managementKey` 为空字符串；密钥存在于 `secureStorage` 加密项。

- 完成标志:
  - 满足 [REQ-001]~[REQ-004]。

### 阶段 2（必备）：登录后快速切换（Header）+（可选）快捷键

- 目标:
  - 在 Header 内提供“切换客户端”入口，登录后可快速切换到另一套 `apiBase/managementKey`。
  - （增强）可启用 `Ctrl+1~9` 快捷键。

- 风险主题:
  - 状态残留与误操作（切错环境）。

- 主要变更:
  - `src/components/layout/MainLayout.tsx`：Header 接入客户端下拉与管理入口；切换时触发重连与刷新。
  - `src/stores/useAuthStore.ts`：补充“切换客户端”能力，避免 401 切换失败触发全局登出。
  - （可选）接入 `useClientKeyboardShortcuts.ts` 作为快捷键入口。

- 验证方式:
  - 切换后强制刷新当前页数据（Header refresh）；确保页面数据来自新环境。
  - 切换失败：不会被登出，不会跳回登录页，连接状态回退到原会话。

- 完成标志:
  - 满足 [REQ-005]。

## 非目标

- 不做客户端配置的云端同步/共享。
