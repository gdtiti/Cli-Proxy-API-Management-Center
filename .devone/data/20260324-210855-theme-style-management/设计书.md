## 方案摘要

- 推荐方案:
  - 将主题状态拆为两层：
    - `themeFamily`: `official | reference`
    - `themeMode`: `auto | light | white | dark`
  - 保留 `resolvedTheme` 作为只读派生值，继续服务现有图标、图表和页面暗色判断。
  - 在 DOM 根节点上同时应用 `data-theme-family` 与现有 `data-theme`，用 CSS variables 和样式分支实现第二套视觉语言。
- 推荐原因:
  - 这个结构能完整保留官方原版主题语义，同时把参考仓库风格做成独立风格族。
  - 风格扩展和明暗扩展被解耦，后续增加第三套风格时不用再改状态模型。
  - 影响范围集中在共享层，业务页面大多不需要改逻辑。

## 备选方案对比

### 方案 A

- 做法:
  - 新增 `themeFamily`，保留当前 `theme` 作为明暗模式，`applyTheme` 同步写入 `data-theme-family` 和 `data-theme`。
  - `themes.scss` 负责 token 覆盖，`layout.scss` / `components.scss` 只在 `reference` 风格族下追加样式分支。
  - 顶栏弹层改成两个分组：风格族选择、明暗模式选择。
- 收益:
  - 架构清晰，向后兼容最好。
  - 官方风格保留度最高。
  - 参考仓库风格可以覆盖到共享布局，不影响业务逻辑。
- 代价:
  - 需要调整 store 类型、持久化结构、菜单 UI 和多份样式文件。
- 风险:
  - 需要明确参考风格下 `white` 模式的处理规则。

### 方案 B

- 做法:
  - 把每个可选项都扁平化成单个 preset，例如 `official-light`、`official-white`、`reference-light`、`reference-dark`。
- 收益:
  - 状态上不存在非法组合。
- 代价:
  - UI 和类型会膨胀，后续接第三套风格需要成倍扩展 preset。
  - 现有代码里与 `theme` 语义相关的地方都要适配新枚举。
- 风险:
  - 逻辑扩散更大，维护成本明显高于方案 A。

## 锁定设计

- 入口模块:
  - `src/types/common.ts`
  - `src/stores/useThemeStore.ts`
  - `src/components/layout/MainLayout.tsx`
  - `src/styles/themes.scss`
  - `src/styles/layout.scss`
  - `src/styles/components.scss`
  - `src/i18n/locales/*.json`
- 关键调用链:
  - `App.tsx` 启动 -> `useThemeStore.initializeTheme()` -> 解析持久化状态 -> 设置根节点属性 -> 全局 SCSS 按属性命中 -> `MainLayout` 从 store 读取当前风格族和主题模式进行展示与切换。
- 数据流:
  - 用户在顶栏选择风格族/主题模式 -> `setThemeFamily` / `setTheme` 更新 Zustand persist 状态 -> `applyTheme` 更新 DOM 属性 -> CSS variables 和样式分支生效 -> 页面继续通过 `resolvedTheme` 获取暗亮信息。
- 错误处理:
  - 对参考仓库风格不支持的 `white` 组合做显式规范化：切换到参考风格时若当前模式为 `white`，自动落到 `light`。
  - 若旧持久化数据没有 `themeFamily`，依赖 Zustand 默认初始值回填为 `official`。
- 配置变更:
  - 不新增新的发布配置。
  - 沿用现有本地存储键 `cli-proxy-theme`，只扩展其状态内容。

- 不追求与参考仓库每个页面像素级一致。
- 不在本轮统一重构所有页面模块的局部 SCSS。
- 不引入运行时主题配置编辑器。
