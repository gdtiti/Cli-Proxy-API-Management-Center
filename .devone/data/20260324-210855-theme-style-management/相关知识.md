# 相关知识

## Docs-First 结论

- 项目知识入口使用 `llmdoc/`，当前仓库没有项目级 `.devone/wiki/`，本轮以 `llmdoc/index.md`、`llmdoc/overview/project-overview.md`、`llmdoc/architecture/frontend-architecture.md` 为主真值源。
- 当前主题入口位于 `src/stores/useThemeStore.ts`，通过 `document.documentElement` 的 `data-theme` 驱动全局样式。
- 当前全局视觉 token 在 `src/styles/themes.scss`，共享布局在 `src/styles/layout.scss`，基础组件在 `src/styles/components.scss`。
- 现有页面大多只依赖 `resolvedTheme === 'dark'` 判断图标或图表暗色方案，因此扩展风格时应保留 `resolvedTheme` 的二值语义。

## 术语与定义

- 风格族:
  - 一组完整视觉语言，包含 token、布局皮肤和基础组件皮肤。
- 主题模式:
  - 当前风格族下的明暗/自动模式，负责决定浅色、纯白、暗色或跟随系统。
- `resolvedTheme`:
  - 页面业务逻辑使用的最终明暗结果，只允许 `light | dark`。

## 调研结论

- 结论 1:
  - 参考仓库风格适合做成第二套风格族，而不是替换官方主题或扁平化 preset 的单项枚举。
- 结论 2:
  - 为了把风格落到真实页面，需要同时改 `themes.scss`、`layout.scss`、`components.scss`，只改 token 不足以还原参考仓库的主题感知。

## 风险假设

- 假设:
  - 参考仓库风格不单独支持 `white`，因此在该风格族下规范化到 `light` 不会违背用户核心诉求。
- 验证方式:
  - 通过 `pnpm type-check`、`pnpm test -- i18n-keys.test.mjs`、`pnpm build` 证明状态模型和构建链路稳定；通过代码审阅确认官方风格分支仍保留。
