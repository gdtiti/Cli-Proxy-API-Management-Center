## 方案摘要

- 推荐方案: 在当前工作区直接执行 `git fetch origin` 与 `git merge origin/main`，按文件逐块解冲突，随后只修复合并后暴露的最小必要类型/构建问题。
- 推荐原因: 用户明确要求在当前工作区完成；该方案能最短路径恢复主线同步状态，同时避免把无关重构混入这次源代码整合任务。

## 备选方案对比

### 方案 A

- 做法: 当前工作区直接 merge，逐块解决冲突并立即验证。
- 收益: 路径最短，和用户要求一致，利于继续后续发布与同步操作。
- 代价: 当前工作区历史更复杂，若记录不完善，后续回看需要额外整理任务文档。
- 风险: 冲突后可能引入局部类型断裂，需要补做最小修复。

### 方案 B

- 做法: 先创建独立 worktree / 分支，再在隔离环境里 merge 与验证。
- 收益: 执行环境更干净，便于回滚和并行验证。
- 代价: 与用户要求冲突，会额外引入分支切换和文档同步成本。
- 风险: 增加流程复杂度，且当前任务本质是一次源代码同步，不值得为此改写执行路径。

## 锁定设计

- 入口模块: Git 合并入口在当前仓库分支，代码修复集中于前端页面、hooks 和 API service。
- 关键调用链: `git fetch origin` -> `git merge origin/main` -> 清理冲突 -> 修复编译错误 -> `pnpm type-check` -> `pnpm build`。
- 数据流: 不新增业务数据流，只恢复 merge 后既有页面与服务模块的编译连通性。
- 错误处理: 发现冲突或编译错误时显式修复，不引入静默 fallback 或假成功路径。
- 配置变更: 无新增配置；构建产物路径保持 `dist/index.html`。
