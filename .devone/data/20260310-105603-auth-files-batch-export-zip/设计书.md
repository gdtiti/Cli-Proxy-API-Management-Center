## 方案摘要

- 推荐方案: 方案 A（复用现有“下载当前筛选”批量 ZIP 导出能力，补齐“导出全部/按标签导出”的产品表达与必要增强）。
- 推荐原因:
  - 当前仓库已存在批量 ZIP 导出主链路（前端拉取 + 浏览器侧打包），实现风险最低、可验证性最好。
  - “按标签导出 / 导出全部”的语义，与现有页面的“类型筛选标签 + 搜索 + 问题过滤”天然对齐，优先做 UX 收敛能最快交付价值。
  - 若用户所说“标签”其实是“自定义 tags”，则需要后端支持；在未确认前不建议直接走跨层改造。

## 备选方案对比

### 方案 A

- 做法:
  - 沿用现有导出实现：`AuthFilesPage` 组装目标文件列表 → `useAuthFilesData.handleBatchDownload()` → `authFilesApi.downloadBlob()` 拉取每个文件 → `createZipBlob()` 打包 → `downloadBlob()` 触发浏览器下载。
  - 产品/交互补齐（按需挑选）：
    - 当筛选为“全部”时，按钮文案从“下载当前筛选”动态变为“下载全部”。
    - 增加一个显式入口（下拉菜单/二级按钮）：“导出全部 / 导出当前筛选（标签）/ 导出问题文件（若已开启问题过滤）”。
    - 可选增强：在底部批量操作栏增加“导出选中”，与“启用/禁用/删除”并列。
  - 可选增强：为批量下载加并发上限（例如 4~8），避免大量文件时瞬时并发过高。
- 收益:
  - 不需要后端改造，交付快。
  - 与现有筛选语义一致，用户学习成本低。
  - 容易写清楚验收口径：导出范围=当前筛选结果（排除 runtimeOnly）。
- 代价:
  - UI 文案与入口需要小幅调整。
  - 并发控制/二次确认属于体验与风险控制，需要额外工作量。
- 风险:
  - 若“标签”被定义为自定义 tags，则方案 A 只能覆盖“类型筛选标签”，无法满足新标签体系需求。

### 方案 B

- 做法: 增加“按多标签导出”弹窗，允许一次勾选多个类型标签（或多个筛选条件组合），生成同一个 ZIP。
- 收益: 解决“一次导出多个类型”的需求，减少来回切换筛选下载。
- 代价: 需要新增 UI 组件与状态管理；导出范围解释更复杂。
- 风险: 复杂度上升，容易引入“用户以为导出了全部但其实只导出了一部分”的理解偏差。

### 方案 C（仅在需要自定义 tags 时）

- 做法: 引入认证文件自定义 tags 的后端存储与 API，前端支持编辑 tags + 按 tags 筛选/导出。
- 收益: 真正支持“自定义标签导出”（如 prod/test/项目分组）。
- 代价: 跨层改造（后端 schema、API、兼容迁移、权限模型），验收成本高。
- 风险: 未确认需求前推进容易返工；同时 tags 的权限与审计策略需要额外设计。

## 锁定设计

- 入口模块:
  - 页面：`src/pages/AuthFilesPage.tsx`（导出按钮位于页面 header actions）
  - 数据/动作：`src/features/authFiles/hooks/useAuthFilesData.ts`
- 关键调用链（现有实现）:
  1. `AuthFilesPage` 计算 `filtered` / `downloadableFilteredFiles`
  2. 点击按钮 → `handleBatchDownload(downloadableFilteredFiles, { archiveName })`
  3. `authFilesApi.downloadBlob(name)` 拉取文件内容（Blob）
  4. `blob.arrayBuffer()` 转为二进制
  5. `createZipBlob(entries)` 生成 ZIP Blob（无第三方依赖）
  6. `downloadBlob({ filename, blob })` 触发浏览器下载
- 数据流:
  - 列表来源：`authFilesApi.list()` → `files`
  - 导出范围：页面筛选（类型标签/type + search + problemOnly）→ `filtered`
  - 导出排除项：`runtimeOnly` 文件在下载前被过滤
- 错误处理:
  - 单文件下载：失败 toast 提示
  - 批量下载：`Promise.allSettled`，成功项写入 ZIP，失败项计数；全失败时报错并不触发下载
- 配置变更:
  - 方案 A（必备部分）不需要新增服务端配置
  - 若加入并发控制，可作为纯前端常量（或可选地读取 visual config）

## 分阶段计划

### 阶段 1

- 目标: 把“按标签导出/导出全部”的语义锁定，并在 UI 上让用户一眼可用。
- 风险主题: 语义风险（标签定义）与误操作风险（导出敏感数据）。
- 主要变更（按需求锁定后选取）:
  - UX：导出按钮文案与入口收敛（全部/筛选/可选：选中）。
  - 风险控制（可选）：导出前二次确认提示“导出的 ZIP 包含敏感凭证”。
  - 性能（可选）：批量下载加入并发上限与进度提示。
- 验证方式:
  - 在本地启动管理端，准备若干认证文件，分别验证：
    - 全部导出
    - 按类型标签筛选导出
    - 开启问题过滤导出（若支持）
  - 人工检查 ZIP 解压后文件数量/文件名/内容是否对应。
- 完成标志:
  - 需求文档中的 REQ-001 ~ REQ-005 可被逐条验证为真。

### 阶段 2

- 目标: 在确认确有需求时，再推进“多标签导出”或“自定义 tags”体系。
- 风险主题: 跨层变更与权限/审计策略。
- 主要变更: 方案 B 或方案 C 的实现。
- 验证方式: 追加对应需求与测试用例，完成端到端验证。
- 完成标志: 新增需求条目验收通过，且不破坏阶段 1 已交付能力。

## 非目标

- 不在 discovery 阶段承诺后端 schema 改造；除非用户明确确认“标签=自定义 tags”且愿意承担跨层实施成本。

## 设计评审结论

- 用户是否批准: 是（2026-03-10）
- 批准范围:
  - 方案 A（阶段 1）：在 `/auth-files` 提供清晰的“导出全部 ZIP”入口，同时保留“导出当前筛选（标签）ZIP”能力。
  - “标签”语义锁定为现有类型筛选标签（`file.type`），不引入自定义 tags 体系。
- 备注:
  - “导出全部”默认语义：导出系统内所有可下载认证文件（排除 `runtimeOnly`），不受当前筛选/搜索/问题过滤影响。
  - “导出当前筛选”语义：导出页面当前筛选结果（类型标签 + 搜索 + 问题过滤组合），排除 `runtimeOnly`。
