## 问题定义

- 当前问题: 管理端需要把“认证文件（auth files）”批量导出到本地，便于备份/迁移/审计。
- 影响对象: 管理端用户（通常是运维/管理员）；导出文件本身包含敏感凭证，必须把“风险认知”和“操作边界”写清楚。
- 触发背景: 需要一次性拿到多份认证文件；逐个下载效率低且易漏；希望按“标签/筛选”导出或导出全部。
- 为什么现在处理: 认证文件数量上来后，手工逐个下载不可持续；同时合规/备份场景需要可重复的导出动作。

## 目标与成功标准

- 目标 1: 用户在 `/auth-files` 页面可一键下载 ZIP，ZIP 内包含目标范围内的认证文件。
- 目标 2: 支持“按标签/筛选导出”和“导出全部”两种模式，且导出结果可预测、可复验。
- 成功标准:
  - 点击导出后浏览器开始下载 `.zip` 文件，文件名能体现导出范围（至少区分全部/筛选/问题文件）。
  - ZIP 内文件数量与页面筛选结果一致（排除 `runtimeOnly` 的临时文件）。
  - 单个文件下载失败时，不阻断其它文件导出；给出“成功/失败数量”的明确反馈。

## 范围边界

### 本次必须完成

- 明确“认证文件标签”的含义，并锁定产品交互（见“风险与待确认项”）。
- 提供用户可理解的导出入口与导出范围说明（避免“其实能导出全部，但用户不知道怎么操作”）。
- 导出 ZIP 不包含 `runtimeOnly` 文件；导出过程具备基本失败反馈（全失败/部分失败）。

### 本次明确不做

- 不做服务端打包 ZIP（本次以“前端拉取 + 浏览器侧打包”为主）。
- 不做导出文件加密/脱敏/二次包装（如需，作为后续增强项）。
- 不做“自定义标签体系”的后端持久化改造，除非用户明确确认“标签”不是现有筛选标签（见备选方案）。

## 验收真相陈述

- [REQ-001] 在 `/auth-files` 页面存在“批量导出 ZIP”的可见入口，点击后浏览器下载 ZIP（非跳转、非后台 silent 失败）。
- [REQ-002] 导出范围支持：
  - 导出全部：包含当前系统内所有可下载认证文件（排除 `runtimeOnly`）。
  - 按标签/筛选导出：至少支持按页面已有筛选维度导出（若“标签”指的是页面上方的类型筛选标签）。
- [REQ-003] ZIP 中不包含 `runtimeOnly` 的运行态临时文件。
- [REQ-004] 批量导出使用“全量并发失败不影响成功项”的策略；若部分失败，仍下载成功项并提示部分失败；若全部失败，给出错误提示且不触发下载。
- [REQ-005] 导出 ZIP 文件名安全化（去除 Windows/浏览器不允许字符），且保证以 `.zip` 结尾。

## 约束与依赖

- 技术约束:
  - 依赖现有单文件下载接口（当前实现为 `GET /auth-files/download?name=...`）。
  - 浏览器侧 ZIP 打包需要控制内存与并发；避免在“认证文件数量极多”时卡死页面。
- 业务约束:
  - 认证文件可能包含敏感凭证；导出行为应被视为高风险操作（至少需要清晰提示，是否加确认弹窗取决于用户策略）。
- 外部依赖:
  - 管理端已登录且具备下载权限。
  - 后端返回的文件名应可用于定位下载资源（以 `name` 为唯一键）。

## 风险与待确认项

- 风险:
  - 安全风险：导出 ZIP 会把敏感凭证落地到用户设备；需要明确这是“用户主动导出”，并考虑是否加二次确认。
  - 性能风险：当前批量导出实现可能对大量文件进行高并发请求；在弱网/大量文件下可能触发浏览器或服务端压力。
  - 语义风险：“标签”指代不清会直接影响设计是否需要后端改造（自定义 tags vs 现有 type 筛选）。
- 待确认项结论（已锁定）:
  - 认证文件“标签”= 页面顶部的“类型筛选标签”（即 `file.type`，如 qwen/kimi/gemini/...）。
  - 若后续确有“自定义 tags 导出”需求，作为阶段 2/方案 C 另行评审，不在本次必备范围内。
