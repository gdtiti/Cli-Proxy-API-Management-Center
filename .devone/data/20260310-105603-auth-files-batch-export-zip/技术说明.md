## 鉴权说明（回答“是否没传 Authorization？”）

- 结论: 批量打包下载与单独下载复用同一套下载方法，不存在“批量漏传 Authorization”这种差异。
- 证据链（代码事实）:
  - 单独下载：`handleDownload()` → `authFilesApi.downloadBlob(name)`
  - 批量导出：`handleBatchDownload()` → 多次调用 `authFilesApi.downloadBlob(name)`
  - `authFilesApi.downloadBlob()` 走 `apiClient.getRaw()`，而 `apiClient` 的请求拦截器会在 `managementKey` 存在时注入 `Authorization: Bearer <managementKey>`。

## 风险、阻塞与未验证项

- 风险:
  - 批量导出目前仍是“把目标文件全部拉取到内存后再打包 ZIP”，当文件数量或文件体积很大时，可能存在浏览器内存压力。
  - 已加入并发上限（默认并发=4）来降低瞬时压力，但在极端数量/体积下仍可能耗时较长；必要时建议按筛选分批导出。
- 阻塞: 无
- 未验证项:
  - 未进行浏览器侧点击实测（需要本地 `pnpm dev` 启动后人工验证下载行为与 ZIP 内容）。
  - 建议人工验收步骤（用于完成 R4 fresh evidence）:
    1. 运行 `pnpm dev`，记录实际监听端口（Vite 默认通常是 `5173`）。
    2. 打开 `/auth-files` 页面，确认列表中存在至少 1 个非 `runtimeOnly` 文件。
    3. 点击“下载全部”，应触发下载 `auth-files-all.zip`；解压后应仅包含非 `runtimeOnly` 文件。
    4. 选择任一类型筛选标签（非 `all`）或启用“仅问题(problemOnly)”或输入搜索关键字：此时应出现“下载当前筛选”。
    5. 点击“下载当前筛选”，应触发下载 `auth-files-... .zip`（文件名会按 `type/problem/filtered` 组合）；解压后文件集合应与当前筛选结果一致。
