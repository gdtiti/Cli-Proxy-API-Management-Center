## 方案摘要

- 推荐方案: 方案 A（直接调用管理端 API 的 Node.js CLI 脚本，限并发导出到本地目录）。
- 推荐原因:
  - 复用现有管理端 API（与 Web UI 同源、同接口），不需要解析页面或做浏览器自动化。
  - 可控性更强：并发、超时、失败重试/不中断、文件名安全化都可以在脚本层明确实现与验收。
  - 更容易避免泄露：脚本强制从环境变量读取管理端 key，不把凭证写入仓库或文档。

## 备选方案对比

### 方案 A

- 做法:
- 写一个 Node.js 脚本（ESM），从 `--base` 计算 API 基址：
  - `GET <base>/v0/management/auth-files` 获取列表；
  - 并发池（默认 4）逐个 `GET <base>/v0/management/auth-files/download?name=...` 下载；
  - 保存到 `--out` 目录（一个文件对应一个本地文件）。
- 认证通过环境变量提供，例如 `CPA_MANAGEMENT_KEY`（不允许硬编码）。
- 收益:
- 实现最直接，依赖最少，鲁棒性高。
- 不受浏览器资源限制影响，便于集成到备份/运维脚本。
- 代价:
- 需要补齐 CLI 参数解析、文件名安全化、并发控制与输出汇总。
- 风险:
- 下载敏感数据落盘的风险必须显式提示，并在默认输出目录/`.gitignore` 上做防呆。

### 方案 B

- 做法:
- 使用 Playwright/浏览器自动化打开 `management.html`，模拟登录并触发页面上的“导出/下载”按钮。
- 收益:
- 复用 UI 行为，不需要理解接口协议。
- 代价:
- 环境要求高（浏览器、驱动、CI 依赖），执行慢且不稳定。
- 需要处理 UI 变化、选择器漂移、下载拦截等问题。
- 风险:
- 自动化过程中更容易意外输出/录屏泄露凭证或导出内容；对运维脚本场景不友好。

## 锁定设计

- 入口模块:
- 新增脚本：`scripts/export-auth-files.mjs`（或 `scripts/export-auth-files.js`，保持 ESM）。
- 运行方式（示例，实际以 README/脚本帮助为准）：
  - `node scripts/export-auth-files.mjs --base http://129.146.34.174:8317 --out <本地目录>`
- 关键调用链:
- `main()` 解析参数 → `resolveApiBase(base)` → `listAuthFiles()` → `downloadPool()` → `writeFileSafely()`
- 数据流:
- 输入：`--base`、`--out`、环境变量 `CPA_MANAGEMENT_KEY`、可选参数（并发/超时/是否包含 runtimeOnly）。
- 输出：本地目录下的认证文件；控制台打印统计与失败列表（必要时写 `report.json`）。
- 错误处理:
- 列表失败：直接退出（非 0 退出码），提示检查 `--base` 与凭证。
- 单文件失败：计入失败统计，继续下载其他文件；最终根据失败数量决定退出码（例如有失败则 exitCode=2）。
- 文件名不安全：跳过并提示（避免写出目录穿越）。
- 配置变更:
- 不改后端；只在仓库中新增脚本与必要的文档说明。
