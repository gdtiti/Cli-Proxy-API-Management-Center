## 测试目标

- 需要被保护的核心行为:
  - 能从用户输入的 `--base`（可为 `http://host:port` 或 `http://host:port/management.html`）正确推导出管理端 API 基址（`/v0/management`）。
  - 认证信息注入逻辑正确：只从环境变量读取，不允许硬编码。
  - 文件名/输出路径安全：永远不会写出到 `--out` 目录之外（防止路径穿越）。
  - 并发池生效：不会一次性并发拉取全部文件。
- 这轮必须覆盖的失败模式:
  - `--base` 非法或无法连接导致列表接口失败。
  - 单个文件下载失败不影响其他文件继续导出（部分失败可重跑）。
  - `name` 含有不安全字符（`..`、`/`、`\\`）被拒绝或安全化处理。

## 单元测试设计

### UT-001

- 目标: `--base` 解析与 API 基址推导正确。
- 覆盖函数/模块: `normalizeApiBase()` / `computeManagementApiBase()`（位于 `scripts/export-auth-files.mjs`）。
- 输入:
  - `http://129.146.34.174:8317`
  - `http://129.146.34.174:8317/management.html`
  - `http://129.146.34.174:8317/v0/management`
  - `129.146.34.174:8317`（若允许无协议输入）
- 预期输出:
  - 统一得到 `http://129.146.34.174:8317/v0/management`
- 失败表现:
  - 输出缺少 `/v0/management`，或错误拼接出重复前缀。

### UT-002

- 目标: 文件名安全化与路径穿越防护正确。
- 覆盖函数/模块: `sanitizeAuthFileName()` / `resolveOutputPath()`（纯函数，位于 `scripts/export-auth-files.mjs`）。
- 输入:
  - 正常：`a.json`、`test@xx.com.json`
  - 恶意：`../a.json`、`..\\a.json`、`a/b.json`、`a\\b.json`
- 预期输出:
  - 正常文件名保持可用（允许 `@`、`.` 等）。
  - 恶意输入不会导致写出 `--out` 目录之外（必要时安全化；若仍越界则抛错）。
- 失败表现:
  - 任何情况下出现输出路径不在 `--out` 目录内。

### UT-003

- 目标: 限并发下载策略生效。
- 覆盖函数/模块: `mapWithConcurrency(items, concurrency, worker)`。
- 输入:
  - 10 个 item，concurrency=2，worker 内部记录“同时进行中的数量峰值”。
- 预期输出:
  - 峰值不超过 2；所有 item 最终都被处理一次。
- 失败表现:
  - 峰值大于并发上限，或部分 item 未被处理。

### UT-004

- 目标: CLI 在 `--quiet` 模式下抑制失败详情，避免在 stdout 打印认证文件名（例如邮箱），降低日志泄露风险。
- 覆盖函数/模块: CLI `main()` 输出逻辑（位于 `scripts/export-auth-files.mjs`），以及失败详情打印分支。
- 输入:
  - 启动本地 server fixture 返回 1 个文件；下载接口返回 500；运行 CLI 并开启 `--quiet`。
- 预期输出:
  - stdout 包含 `Summary` 与失败计数，但不包含具体认证文件名。
- 失败表现:
  - stdout 打印了具体认证文件名，或未给出失败计数/汇总信息。

## 属性测试设计

### PT-001

- 性质: 任何 `name` 都不应导致写出 `--out` 目录之外。
- 输入空间:
  - 随机字符串（包含 `/`、`\\`、空白、控制字符、unicode 等）
  - 特殊案例：`..`、`.`、`CON`、`NUL`（Windows 保留名）
- 不变量:
  - 若返回可用文件名，则：
    - 不包含 `/` 或 `\\`
    - 不包含 `..`
    - 最终 `path.resolve(outDir, filename)` 必须以 `outDir` 为前缀
- 预期失败模式:
  - 对极端输入应选择“拒绝并报错/跳过”，而不是尝试写文件。
