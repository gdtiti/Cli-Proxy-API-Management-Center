# 技术设计文档：cached-api-clients

## 上下文

CLI Proxy API 管理中心当前每次连接都需要用户手动输入服务器地址和管理密钥。现有的 `useAuthStore` 已支持基本的 `rememberPassword` 功能，但只能保存单个用户的凭据。当用户需要同时管理多个环境（本地、测试、生产）时，频繁切换非常不便。

### 当前状态

- 使用 Zustand + persist 中间件存储认证状态
- 支持 `apiBase`、`managementKey` 和 `rememberPassword` 字段
- 使用 `localStorage` 作为持久化存储（通过 zustand 的 createJSONStorage）
- 登录页面组件：`src/pages/LoginPage.tsx`

### 约束

- 后端无需修改，纯前端功能
- 管理密钥必须安全存储（考虑复用现有的 secureStorage）
- 保持与现有登录流程的兼容性

## 目标 / 非目标

**目标：**

- 实现多客户端配置本地缓存功能
- 提供快速切换客户端的 UI（支持下拉选择和快捷键）
- 支持添加、编辑、删除客户端配置
- 配置持久化存储，重启后保持

**非目标：**

- 不实现客户端间的数据同步
- 不实现服务端配置管理
- 不改变现有单用户登录流程的核心逻辑

## 决策

### 1. 存储方案选择

**选项 A：扩展现有 useAuthStore**

- 优点：与现有代码集成度高，复用现有 persist 逻辑
- 缺点：职责混杂，store 会变得臃肿

**选项 B：创建独立的 useClientCacheStore**（推荐）

- 优点：职责分离清晰，易于维护和测试
- 缺点：需要新增存储逻辑

**决策**：选择选项 B，创建独立的 store 管理客户端缓存。

### 2. 密钥存储策略

**选项 A：localStorage 明文存储**

- 优点：实现简单
- 缺点：安全风险高

**选项 B：复用 secureStorage 服务**

- 优点：利用现有加密机制，安全性高
- 优点：密钥统一管理
- 缺点：需要调用现有服务

**决策**：选择选项 B，复用 `src/services/storage/secureStorage.ts`。

### 3. UI 组件方案

**选项 A：独立客户端管理页面**

- 优点：功能完整，管理方便
- 缺点：需要新增路由和页面

**选项 B：在登录页集成下拉选择**（推荐）

- 优点：用户操作路径最短，切换最方便
- 缺点：登录页代码可能变复杂

**决策**：选择选项 B，登录页增加客户端选择下拉菜单。同时可以提供独立的设置入口管理多个客户端。

### 4. 快捷键方案

使用键盘快捷键（如 `Ctrl+1` 到 `Ctrl+9`）快速切换到对应的客户端。使用 React 的 useEffect 绑定全局键盘事件。

## 风险 / 权衡

| 风险                 | 缓解措施                       |
| -------------------- | ------------------------------ |
| 客户端配置数据丢失   | 提供导出/导入功能              |
| 密钥泄露风险         | 使用 secureStorage 加密存储    |
| 快捷键与浏览器冲突   | 提供开关选项，可禁用快捷键     |
| 存储配额限制         | 限制最大客户端数量（如 20 个） |
| 过期凭据导致连接失败 | 切换时自动验证，失败时提示用户 |
